# Интеграция с NIST STS 2.1.2

Проект использует **официальный пакет статистических тестов NIST STS 2.1.2**
(NIST SP 800-22), который выпускается NIST в общественное достояние. Сам код
не входит в репозиторий дипломной работы — он просто слишком объёмный, и каждая
сборка под конкретную архитектуру лучше воспроизводится локально.

## Сборка из исходников (Linux)

```bash
cd nist_sts
wget https://csrc.nist.gov/CSRC/media/Projects/Random-Bit-Generation/documents/sts-2_1_2.zip
unzip sts-2_1_2.zip                              # появится sts-2.1.2/
cd sts-2.1.2
make                                              # бинарник: sts-2.1.2/assess
```

В некоторых дистрибутивах `make` может ругаться на `-O3`/`-Werror`. Тогда
правьте `Makefile`:

```makefile
CFLAGS = -c -Wall -O2 -Wno-unused-result
```

Проверка работоспособности:

```bash
echo "0101010101010101..." | ./assess 1000000     # просто чтобы увидеть меню
```

## Запуск тестов

Скрипт `run_sts.sh` подаёт бинарный файл `--in` в `assess` пакетным режимом
со стандартными параметрами:

- длина битовой последовательности **n = 1 000 000** (миллион бит на блок);
- число блоков (streams) — `--streams` (по умолчанию 10, как в SP 800-22);
- выбираются **все 15 тестов**;
- результаты собираются из `experiments/AlgorithmTesting/finalAnalysisReport.txt`
  и парсятся `parse_results.py`.

Пример:

```bash
bash run_sts.sh \
    --sts ./sts-2.1.2 \
    --in  ../data/processed/02_zener/run_001_lsb1.bin \
    --out ../data/reports/02_zener/nist \
    --streams 10
```

После запуска появятся:

- `nist/finalAnalysisReport.txt`   — оригинальный отчёт NIST;
- `nist/results.csv`               — таблица p-value/proportion для 15 тестов;
- `nist/results.md`                — то же в markdown (для report.md).

## Интерпретация

NIST STS принимает гипотезу случайности при выполнении ОБОИХ условий:

1. **p-value**: для распределения p-value по блокам **uniformity p_T ≥ 1e-4**.
2. **proportion**: доля блоков, прошедших тест, должна попасть в интервал
   **p̂ ± 3·√(p̂(1-p̂)/m)**, где p̂ = 1−α = 0.99.

Скрипт `parse_results.py` считает оба критерия и помечает каждый тест как
**PASS** / **FAIL**. Тестов, у которых p_value = 0 ровно (assess пишет `* * *`),
помечаются как FAIL.

## Альтернативы (если STS не собирается)

В контрольной точке можно прогнать pure-Python вариант — `dieharder` или
`testu01`. Но для дипломной работы важно именно соответствие SP 800-22, поэтому
официальный STS обязателен.

## Что писать в дипломную работу

Из `results.md` берётся таблица NIST STS — она аналогична таблице из
приложения 1 дипломной работы, только с реальными p-value и колонкой "пройдено/не
пройдено". Импортируется в `parts/05-chapter2.tex` как новый раздел
"Результаты NIST STS для источника N".
